# Nexor — Data Model v0.1

## 1. Purpose

This document defines the initial data model for Nexor. Its goal is to support the operational core of the system in local-first mode, covering log ingestion, job normalization, roll assembly and closure, export, and auditable historical consultation.

This version focuses on the minimum model required to start implementation with consistency between domain, persistence, and interface.

## 2. Scope

### Included in v0.1

* operational core
* minimum master data
* basic historical search
* auditability for the main operational flow

### Out of scope for v0.1

* detailed planning
* detailed inventory control
* advanced analytics
* online sync
* automated backup
* multi-station coordination

## 3. Official Vocabulary

The following terms must be treated as official across documentation, code, persistence, and interface:

* **Log**: raw imported production record
* **Job**: normalized operational event derived from a log
* **Roll**: consolidated operational unit used for closure, export, and historical consultation
* **Machine**: source machine reference
* **Operator**: operator reference
* **Fabric**: normalized fabric reference
* **Fabric Alias**: alternate input name mapped to an official fabric
* **Historical Search**: consultation of previously closed rolls
* **Audit**: traceability of changes, origins, exports, and corrections
* **Local-First**: primary execution and persistence mode in local environment
* **Hybrid Evolution**: future online/backend expansion without breaking the local core

## 4. Modeling Principles

The Nexor data model must follow these principles:

1. **Domain before interface**
   Business entities and rules must exist independently from the UI.

2. **Explicit states in data**
   Operational states must be represented in the model, not inferred only from screens.

3. **Local database as initial source of truth**
   In v0.1, persisted local data is authoritative.

4. **Traceability from the beginning**
   Imported origin, corrections, exports, and reopen actions must remain auditable.

5. **Exports generated from persisted data**
   PDF and JPG mirror outputs must be generated from stored `Roll` + `Job` data, never from transient UI state.

6. **Controlled scope growth**
   Entities related to planning, inventory, analytics, or online sync must not distort the operational core model in v0.1.

## 5. Entities Covered in v0.1

### Core Entities

* `Log`
* `Job`
* `Roll`

### Minimum Supporting Entities

* `Machine`
* `Operator`
* `Fabric`
* `FabricAlias`

## 6. Role of Each Entity

### `Log`

Represents a raw record imported from source files generated by the production environment.

Its main responsibilities are:

* preserve the original imported payload
* support deduplication
* provide traceability of origin
* serve as the source for job normalization

### `Job`

Represents a normalized operational event derived from a valid `Log`.

Its main responsibilities are:

* preserve the original job name and normalized name
* store the production metrics used by Nexor
* support review and correction flows
* allow assignment into a roll draft

### `Roll`

Represents the consolidated operational unit used for closure, export, consultation, and audit.

Its main responsibilities are:

* group multiple jobs
* freeze a validated composition on closure
* provide a stable base for export and re-export
* support historical consultation and review

### `Machine`

Represents the source machine associated with jobs and rolls.

### `Operator`

Represents the operator reference associated with jobs and rolls when available.

### `Fabric`

Represents the official normalized fabric catalog entry.

### `FabricAlias`

Represents alternate textual inputs that map to a normalized `Fabric`.

## 7. Core Relationships

The following relationships define the operational core of the system:

* one `Log` may originate zero or one `Job`
* one `Job` belongs to zero or one `Roll`
* one `Roll` aggregates many `Job`s
* one `Job` may reference one `Machine`
* one `Job` may reference one `Operator`
* one `Job` may reference one `Fabric`
* one `Fabric` may have many `FabricAlias` entries

### Relationship Notes

* A `Log` may not generate a `Job` if it is invalid, duplicated, or intentionally ignored.
* A `Job` must never silently belong to more than one `Roll`.
* Reassignment of a `Job` from one `Roll` to another must be explicit and auditable.

## 9. States and Transitions

### 9.1 `Log` States

* `new`
* `parsed`
* `invalid`
* `duplicated`
* `ignored`
* `converted`

### 9.2 `Job` States

* `pending_review`
* `ready`
* `suspicious`
* `assigned_to_roll`
* `ignored`
* `corrected`

### 9.3 `Roll` States

* `draft`
* `closed`
* `exported`
* `reviewed`
* `reopened`

### Transition Rules

* a `Log` cannot become `converted` before successful parsing
* an `invalid` log must preserve raw origin and error detail
* a `Job` should become `assigned_to_roll` only after explicit inclusion in a draft
* a `Roll` in `draft` may still change composition
* a `Roll` in `closed` must freeze composition
* `exported` indicates that at least one formal output was generated
* `reopened` must preserve evidence of prior closure/export cycle

## 10. Invariants and Business Rules

The following rules must hold in the model:

1. `fingerprint` must prevent silent duplicate import.
2. `raw_payload` must never be discarded after import.
3. `actual_printed_length_m = height_mm / 1000`.
4. `vpos_mm` is an offset value and must not be added to real printed length.
5. a `Roll` cannot be closed without at least one `Job`.
6. closing a `Roll` freezes its composition.
7. reopening a `Roll` must be auditable.
8. exports must be generated from persisted roll composition.
9. historical consultation must not depend on a reconstructed transient UI state.

## 11. Audit and Traceability

The model must preserve enough information to support audit and historical consultation.

### Minimum audit data

* creation and update timestamps
* import source reference
* parsing error details when applicable
* correction notes
* reassignment history when relevant
* reopen history
* export paths and export timestamps

## 12. Persistence and Boundaries

### Persistence Strategy

* local embedded database is the default persistence mode for v0.1
* repositories should be separated by core entity
* exported artifacts must be stored outside the model layer and referenced by path or metadata

### Boundaries

* domain entities must not depend on UI structures
* parsing and file system access belong to infrastructure
* export generation belongs to infrastructure/application services
* interface must consume persisted state, not define business truth

## 13. Data Flow Supported by the Model

The v0.1 model must support the following operational sequence:

1. import raw logs
2. identify and deduplicate
3. parse source content
4. persist `Log`
5. normalize into `Job`
6. assemble `Roll` draft
7. close `Roll`
8. export outputs
9. consult and re-export historical rolls

## 14. Explicitly Out of Scope in v0.1

The following entities or domains may exist later but are not part of the operational core model in this version:

* `InventoryItem`
* `ProductionPlan`
* advanced analytical metrics
* remote backend sync structures
* automated backup structures
* multi-station orchestration entities

## 15. Open Decisions

The following points remain intentionally open and should be resolved before implementation reaches later stages:

* final language for code identifiers
* official `roll_code` rule and format
* when `operator_id` becomes mandatory
* whether suspicious jobs may enter a roll with explicit override
* how to resolve ambiguous fabric aliases
* whether machine is inferred, selected, or both depending on source quality
